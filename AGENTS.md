# 协作规范

## 1. Git 提交规范
- 每次修改必须提交 git，保持工作区整洁
- 提交标题格式：`[English]: 中文标题`
- 常用动作：`Add`、`Fix`、`Refactor`、`Docs`、`Style`、`Test`、`Chore`
- 提交正文与标题换行分隔，使用中文 Markdown 记录修改范围、核心变更、兼容性影响

## 2. AGENTS.md 修改规范
- 修改必须经过用户确认，不可随意更改

## 3. 文档体系与阅读规则

### 持久化文档（每轮必读，跨任务稳定）

以下文件是项目的行为准则和长期边界，Agent 在每轮对话开始时必须加载理解：

| 优先级 | 文件 | 为什么必读 |
|--------|------|-----------|
| **P0** | `AGENTS.md`（本文件） | 协作规范、代码风格、AI 角色定义 |
| **P0** | `docs/internal/AI记忆文档.md` | 项目定位、入口边界、构建命令、已知问题、ModularUI2 集成与陷阱 |
| **P0** | `docs/errors/README.md` | 已发生过的错误与教训，避免重复踩坑 |

### 参考文档（涉及相关领域时读取）

| 文件 | 何时读取 |
|------|----------|
| `docs/README.md` | 文档导航入口 |
| `docs/singularity-me-overview.md` | 需要理解模组整体功能、设备清单、设计理念 |
| `docs/singularity-me-architecture-whitepaper.md` | 深入架构细节（网络模型、生命周期、跨维度机制） |
| `docs/compat-profile.md` | AE2 生态兼容（AE2FC、WCT 等可选依赖） |

### 更新规则

- 开始开发前，必须先阅读 `docs/internal/AI记忆文档.md`
- 目录结构、主入口、脚本命令、运行方式、能力边界变化 → 同步更新 `docs/internal/AI记忆文档.md`
- 阶段目标、进度状态、下一步执行项变化 → 同步更新 `docs/internal/AI记忆文档.md`
- 发现错误、教训 → 写入 `errors/README.md`
- 只记录长期稳定信息，不承担流水账
- AI记忆文档不承担：临时调试记录、一次性试验结论、阶段流水账、修复日志、类清单、示例页细节、版本兼容试验记录等易膨胀内容
- 能归属到具体文档的信息写回原文档，不复制到 AI记忆文档
- 更新时优先删减归并，避免按日期持续追加；除非属于跨任务长期稳定边界，否则不新增条目

## 4. 代码规范

### 命名规范
- 类名：使用英文大驼峰命名法
- 方法名：使用英文小驼峰命名法
- 变量名：使用英文小驼峰命名法
- 常量名：使用英文全大写，下划线分隔

### 注释规范
- 类与方法必须中文注释，重要方法加 Javadoc
- 字段、成员变量、临时变量按可读性需要选择性注释，不强制逐一注释
- 复杂逻辑添加行内注释，解释实现思路
- 注释应简洁明了，避免冗余
- 修改代码时同步更新相关注释

### 代码格式
- 4 空格缩进
- 左大括号不换行，右大括号独占一行
- 运算符两侧添加空格
- 逗号、分号后添加空格

### 其他规范
- 异常处理：捕获具体异常，避免捕获通用异常
- 代码复用：提取重复代码为独立方法

## 6. 错误记录规范
- 发现错误、反复出现的问题、曾经做错的事情、重要教训必须及时记录，避免同一错误重复发生
- 索引文件 `docs/errors/README.md` 作为"指针文档"，使用二级标题（`## [日期或编号]-[错误简述]`）指向 `docs/errors/` 下的具体错误分析文档
- 详细文档命名建议：`ERROR-[日期]-[简述].md`（如 `ERROR-20260418-git-commit.md`）
- 每条记录至少包含：错误现象、触发场景、根本原因、修复方案、预防措施
- 目的是帮助后续 Agent 快速查阅历史问题，快速排错，避免同样错误反复找原因
